# Phase 19.1 Plan 03: Per-Job LLM Config (Gap-2 Fix) Summary

**One-liner:** Per-job LLM model routing wired into all three BullMQ handlers by deriving the 5 config-dependent components from `job.config.llm` over the shared recorder-equipped `llmClient` on each task invocation.

## What Was Built

### `packages/queue/src/derive-job-deps.ts`

Two exported functions:

**`deriveJobDeps(base, llmClient, jobLlmConfig, jobId)`**

Constructs a new `WorkerDeps` from a job's `LLMConfig` over the shared `llmClient`. The 5 LLM-config-dependent components are rebuilt:

```typescript
classifier: new PageClassifier(llmClient, jobLlmConfig);
extractor: new StaticExtractor(llmClient, jobLlmConfig, jobId);
schemaEvolver: new SchemaEvolverImpl(llmClient, jobLlmConfig);
reconciler: new DataReconcilerImpl(llmClient, jobLlmConfig);
linkEvaluator: new LLMLinkEvaluator(llmClient, resolveModel(jobLlmConfig, 'linkEvaluation'));
```

All other fields (crawler, repos, contentStore, queues, eventPublisher, robotsChecker, etc.) are passed through as shared singletons. The shared `llmClient` carries the `AlsUsageRecorder` from Plan 02, so per-job usage attribution continues to work — the recorder reads `tenantId/jobId` from ALS, not from the client instance.

**`resolveJobDeps(deps, sharedClient, jobId, tenantId)`**

Async handler helper:

- If `sharedClient` is `undefined` → returns `deps` unchanged (test-injection path: no crash)
- Otherwise: loads `job = await deps.jobRepo.findById(jobId, tenantId)`, reads `job?.config?.llm` (with fallback to `deps.defaultLlmConfig` then `'deepseek/deepseek-v4-pro'`), and calls `deriveJobDeps`

### `packages/queue/src/worker-entrypoint.ts`

One new block after the usage recorder wire-up (Plan 02):

```typescript
// Per-job-derivation seams (Plan 03)
if (built?.llmClient) {
  (deps as any).llmClient = built.llmClient;
  (deps as any).defaultLlmConfig = built.llmConfig;
}
```

This threads the shared CircuitBreaker-wrapped `llmClient` and the construction-time default `llmConfig` onto `deps` as plain properties. The guard (`built?.llmClient`) means this is a no-op when `_opts.deps` was passed (test-injection path), keeping the test suite clean.

### Handler Changes (crawl / schema / reconciliation)

Each handler begins with:

```typescript
const sharedClient = (deps as any).llmClient as LLMClient | undefined;
const jobDeps = await resolveJobDeps(deps, sharedClient, jobId, tenantId);
```

Then passes `jobDeps` into the orchestrator call. The pre-crawl checks in `crawl-worker.ts` (tenant quota, page budget, robots.txt, rate limiter) still read from base `deps` — they don't touch LLM components, and `jobDeps.crawler === deps.crawler` (singleton) anyway.

### `packages/queue/src/derive-job-deps.test.ts`

6 unit tests covering the four plan-specified behaviors:

1. **New instances returned** — derived extractor/classifier/schemaEvolver/reconciler/linkEvaluator are not the base stubs
2. **linkEvaluator model** — built with `resolveModel(config, 'linkEvaluation')` = `primaryModel` when no override
3. **Singletons preserved** — crawler/contentStore/jobRepo/taskRepo/queues/dbPool/etc are the same object references
4. **modelOverrides honored** — extractor's config carries the extraction override; linkEvaluator falls back to primaryModel
5. **resolveJobDeps no-client fallback** — returns base deps unchanged when `sharedClient` is `undefined`
6. **resolveJobDeps with client** — loads job.config.llm, derives new deps with the job's model

## How the Shared llmClient Is Threaded onto Deps

```
buildWorkerDeps()
  → returns { deps, rawClient, llmClient, llmConfig }

worker-entrypoint.ts (after Plan 02 usage recorder wire-up):
  (deps as any).llmClient = built.llmClient;      // CircuitBreaker-wrapped
  (deps as any).defaultLlmConfig = built.llmConfig; // worker default

crawl-worker.ts / schema-worker.ts / reconciliation-worker.ts:
  const sharedClient = (deps as any).llmClient as LLMClient | undefined;
  const jobDeps = await resolveJobDeps(deps, sharedClient, jobId, tenantId);
```

The `(deps as any)` cast avoids widening the `WorkerDeps` public type, keeping the public API stable for Plan 04.

## The No-Client Fallback (Why Existing Tests Stay Green)

The existing crawl/schema/reconciliation worker unit tests call `processCrawlJob(data, deps)` where `deps` is a synthetic mock built by `createMockDeps()`. These mocks don't set `(deps as any).llmClient` because `worker-entrypoint.ts` sets it only when `built?.llmClient` is truthy (the `buildWorkerDeps` path). So:

1. `sharedClient` is `undefined`
2. `resolveJobDeps` returns `deps` unchanged
3. All 31 crawl-worker tests, 7 schema-worker tests, and 4 reconciliation-worker tests continue to pass against the mock deps

## Deviations from Plan

None — plan executed exactly as written.

The `resolveJobDeps` fallback chain (`job?.config?.llm ?? deps.defaultLlmConfig ?? {primaryModel: 'deepseek/deepseek-v4-pro'}`) matches the plan spec exactly. The attachment of both `llmClient` and `defaultLlmConfig` onto deps (rather than just `llmClient`) was explicitly specified by the plan as the recommended approach.

Combined with Plan 02's ALS usage recorder, the `llm_usage` rows will carry the correct per-job model name.

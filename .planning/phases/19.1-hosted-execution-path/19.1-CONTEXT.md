# Phase 19.1: Hosted Execution Path Completion - Context

**Gathered:** 2026-06-12
**Status:** Ready for planning
**Source:** Live discovery during the 19-05 Render deploy + 19-09 sizing smoke (this session). This CONTEXT _is_ the research — every claim below was confirmed by code-read and/or a live 3-page crawl smoke that DLQ'd with `WorkerDeps not initialized`.

<domain>
## Phase Boundary

Make the **hosted/queued execution path actually process crawls end-to-end**. Today the BullMQ worker — both the standalone `Dockerfile.worker` (`CMD packages/queue/dist/worker-entrypoint.js`) and the Render **embedded** worker (`SPATULA_EMBEDDED_WORKER=1` in `apps/api`) — starts its queue consumers (7 queues + heartbeat) but throws `WorkerDeps not initialized` on every job because `new WorkerDeps(...)` is constructed **nowhere** in production code. The only complete crawl path is the CLI local mode (`apps/cli/src/commands/run.ts` → `LocalPipelineRunner`), which builds per-job deps for the single project crawl.

**In scope (the three coupled gaps + verification):**

1. Build `WorkerDeps` in `startWorker()` so the worker processes jobs.
2. Apply each job's LLM config (model tier) to that job's LLM calls (per-job, race-safe).
3. Record every worker LLM call to `llm_usage` with correct tenant + job (race-safe), so `/api/v1/usage` cost works and sizing can measure.
4. Re-verify DEPLOY-02 (19-05) **live** with a real crawl, and confirm DEPLOY-09 (19-09) sizing harness runs green (non-zero, tier-distinct cost/page).

**Out of scope:** filling the literal CX32 measured sizing table (that stays DEPLOY-09's manual live-run checkpoint); k8s changes beyond what's needed for the worker to function; any SaaS/billing concerns (carved out).

**Groundwork ALREADY committed** (commits `b76cdf2`): `docker-compose.prod.yml` DB/Redis host overrides + configurable `${SPATULA_PG_PORT}`/`${SPATULA_REDIS_PORT}`/`${SPATULA_API_PORT}`; `GET /api/v1/jobs/:id` surfaces `stats.pagesCompleted`; `scripts/sizing-baseline.ts` rebuilt pure-HTTP. Plans should BUILD ON these, not redo them.
</domain>

<decisions>
## Implementation Decisions

### Gap 1 — Worker DI (EXEC-01)

- **Where:** `packages/queue/src/worker-entrypoint.ts`, function `startWorker(_opts?: { deps?: WorkerDeps })` (line ~47). It declares `let deps: WorkerDeps | undefined` (~line 80) and checks `if (!deps) throw new Error('WorkerDeps not initialized')` in every handler (lines 86/110/127/144/209) but NEVER assigns `deps`. The `_opts.deps` param is ignored.
- **Fix:** Construct a full `WorkerDeps` inside `startWorker()` (after the DB pool + queues are created, BEFORE the workers can process) and assign it to `deps`. Honor an injected `_opts.deps` first (for tests), else build the default.
- **WorkerDeps shape** (`packages/queue/src/worker-deps.ts` — `WorkerDepsConfig`): `dbPool, crawler, extractor, classifier, contentStore, schemaEvolver, reconciler, jobRepo, taskRepo, pageRepo, extractionRepo, schemaRepo, entityRepo, sourceTrustRepo, entitySourceRepo, exportRepo, actionRepo, queues` (required) + optional `eventPublisher, linkEvaluator, robotsChecker, rateLimiter, pageBudget, completionChecker, tenantRepo`.
- **Construction recipe** (mirror the two existing compositions):
  - Repos + contentStore + dbPool + queues: copy `apps/api/src/main.ts` `buildAppDeps()` pattern — `new JobRepository(db)`, `CrawlTaskRepository`, `PageRepository`, `ExtractionRepository`, `SchemaRepository`, `EntityRepository`, `EntitySourceRepository`, `SourceTrustRepository`, `ExportRepository`, `ActionRepository`, `TenantRepository`; `contentStore` = `PgContentStore(db)` (or S3 via `createContentStore` when `CONTENT_STORE=s3`, same env switch as main.ts). `createDatabasePool()` already called in startWorker — reuse its `db`/`pool`.
  - LLM-dependent components: copy `apps/cli/src/commands/run.ts` (steps 7–11, lines ~130–207) — `rawClient = createLLMClient({ provider: 'openrouter', openrouter: { apiKey: process.env.OPENROUTER_API_KEY } })`; wrap `llmClient = new CircuitBreakerLLMClient(rawClient)`; `classifier = new PageClassifier(llmClient, llmConfig)`; `extractor = new StaticExtractor(llmClient, llmConfig, tenantId)`; `schemaEvolver = new SchemaEvolverImpl(llmClient, llmConfig)`; `reconciler = new DataReconcilerImpl(llmClient, llmConfig)`; `linkEvaluator = new LLMLinkEvaluator(llmClient, resolveModel(llmConfig, 'linkEvaluation'))`; `robotsChecker = new RobotsTxtChecker()`; `rateLimiter = new InMemoryDomainRateLimiter()`; `completionChecker = new CrawlCompletionChecker()` (from `packages/core/src/crawlers/completion-checker.ts`).
  - Crawler: `crawler = await CrawlerFactory.create({ type: process.env.SPATULA_CRAWLER ?? 'playwright', firecrawlApiKey: process.env.FIRECRAWL_API_KEY })`. **Caveat (document, do not silently ignore):** the distroless `Dockerfile.worker` ships NO Playwright browser, so `playwright` only works where a browser is present (embedded worker on a host/VM with Playwright, or the CLI image). For the distroless worker image, `firecrawl` (needs `FIRECRAWL_API_KEY`) is the viable crawler. Decision: default `playwright`, override via `SPATULA_CRAWLER`; log the chosen crawler at startup; the runbook/notes must state the distroless-worker→Firecrawl requirement. (A Playwright-in-distroless image change is a separate, out-of-scope follow-up — log it if not addressed.)
- **Fail-loud:** if `OPENROUTER_API_KEY` is missing, throw a clear error at startup (the worker cannot extract without it) — do NOT silently start a deps-less worker.
- **The `llmConfig` problem:** the components above take an `llmConfig` at construction, but the worker has no single job config. See Gap 2 — the base deps should be built with a sensible default `llmConfig` (e.g. `{ primaryModel: process.env.SPATULA_DEFAULT_MODEL ?? 'deepseek/deepseek-v4-pro' }`), and per-job config is applied per Gap 2.

### Gap 2 — Per-job LLM config (EXEC-02)

- **Why:** `StaticExtractor` (and classifier/schemaEvolver/reconciler/linkEvaluator) resolve the model via `resolveModel(this.config, 'extraction')` where `this.config` is the **construction-time** `llmConfig`. The worker builds these once → each job's `llm.primaryModel` (the sizing tiers `fast`/`primary`/`smart`) would be ignored. Confirmed: `packages/core/src/extraction/static-extractor.ts:76,105`.
- **Decision (recommended):** derive **per-job** LLM-config-dependent components inside each worker handler. Each handler already has `job.data.jobId` + `tenantId`; load the job (`deps.jobRepo.findById(jobId, tenantId)`), read `job.config.llm`, and construct a per-job `extractor`/`classifier`/`schemaEvolver`/`reconciler`/`linkEvaluator` from that config, reusing the shared `llmClient`, `crawler`, repos, and `contentStore`. Pass a per-job `WorkerDeps` (base deps with the config-dependent components overridden) into `processCrawlJob`/`processReconciliationJob`/etc.
  - Keep it cheap: only the 5 config-dependent components are rebuilt per job (they're thin wrappers over the shared `llmClient`); the crawler/repos/contentStore/queues stay shared singletons.
  - This is race-safe by construction (each job has its own component instances; no shared mutable model state).
- **Alternative considered (note, don't default to):** thread per-call config through `LLMCompletionRequest` — larger change to the LLM interface + every call site; rejected as too invasive for this phase.

### Gap 3 — LLM usage recording (EXEC-03)

- **Why:** `OpenRouterClient.setUsageRecorder(recorder)` exists and `complete()` fires `this.usageRecorder.record({model, promptTokens, completionTokens, totalTokens, costUsd, durationMs})` — but NOTHING in production calls `setUsageRecorder`. `DefaultUsageRecorder` (apps/api) binds `tenantId`/`jobId` at construction. Confirmed: `llm_usage` is always empty; `/api/v1/usage` reads an empty table.
- **Decision:** **AsyncLocalStorage** for race-safe per-job attribution with a shared worker client:
  1. Add an ALS store (in `@spatula/queue`) of `{ tenantId, jobId }`.
  2. Implement an `LLMUsageRecorder` (the core interface) that reads `{tenantId, jobId}` from the ALS store and writes to `LlmUsageRepository.insert({ tenantId, jobId, model, promptTokens, completionTokens, totalTokens, costUsd: costUsd.toFixed(6), purpose })` (mirror `apps/api/src/services/usage-recorder.ts` `DefaultUsageRecorder.record`, but pull tenant/job from ALS instead of the constructor). Fire-and-forget with a `.catch` warn (same as DefaultUsageRecorder).
  3. Set this recorder ONCE on the **raw** `OpenRouterClient` BEFORE wrapping with `CircuitBreakerLLMClient` (the wrapper may not expose `setUsageRecorder`; set it on the raw client, then wrap). Guard with `if ('setUsageRecorder' in rawClient)`.
  4. Wrap each LLM-using worker handler body in `als.run({ tenantId, jobId }, () => processX(...))` so concurrent jobs each see their own context. Handlers to wrap: crawl, schema-evolution, reconciliation (the LLM-touching ones).
- **Result:** every worker LLM call records to `llm_usage` with correct tenant+job; `/api/v1/usage` `byJob`/`byModel` reflect real spend; sizing reads per-tier cost via distinct job IDs.

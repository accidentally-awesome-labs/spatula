<objective>
Ship zero-downtime API-key rotation (AUTH-05). Add `POST /api/v1/api-keys/:id/rotate` that, in a single DB transaction, generates a new key inheriting the original's scopes verbatim and stamps the OLD key with `expiresAt = now + graceSeconds` so both keys validate during the grace window — clients adopt the new key, verify it works, then let the old one age out. `graceSeconds` defaults to `86400` (24h), accepts a `0..604800` request override, and is server-clamped to the 7-day cap. Rotation emits an `api_key.rotated` audit event with both ids and returns the D-16 response shape.

Purpose: Operators must rotate compromised or aging keys without dropping in-flight requests. The two-key grace window (AWS IAM rotation UX) makes that zero-downtime by construction.
Output: A `rotate()` method on `ApiKeyRepository`, a rotate-request schema, the rotate route handler in `api-keys.ts`, and an integration test proving grace-window validity and post-grace expiry.
</objective>

    Add a `rotate()` method to `ApiKeyRepository` in `packages/db/src/repositories/api-key-repository.ts`. Implement per RESEARCH Pattern 4 / D-14 / D-15 / D-16.

    Signature: `async rotate(keyId: string, tenantId: string, newKeyMaterial: { keyHash: string; keyPrefix: string }, graceSeconds: number): Promise<{ oldKey: <row>; newKey: <row> }>`.
    (The route handler generates the raw key + hash + prefix — the repo only receives the hash material so the raw key never touches the repo layer.)

    Inside a single `this.db.transaction(async (tx) => { ... })`:
    1. SELECT the original row: `tx.select().from(apiKeys).where(and(eq(apiKeys.id, keyId), eq(apiKeys.tenantId, tenantId)))`.
    2. If no row → throw `new StorageError('API key ' + keyId + ' not found', { context: { keyId, tenantId } })`.
    3. If `orig.revokedAt` is set → throw `new StorageError('API key ' + keyId + ' is already revoked and cannot be rotated', { context: { keyId } })`.
    4. Compute `const graceUntil = new Date(Date.now() + graceSeconds * 1000)`.
    5. INSERT the new key: `tx.insert(apiKeys).values({ tenantId, keyHash: newKeyMaterial.keyHash, keyPrefix: newKeyMaterial.keyPrefix, name: orig.name + ' (rotated)', scopes: orig.scopes, supersedes: orig.id, supersededExpiresAt: graceUntil }).returning()` — `scopes` copied verbatim from `orig.scopes` (D-15). Note `supersededExpiresAt` is stored on the NEW row so the response can return it without a second query (D-16).
    6. UPDATE the old key: `tx.update(apiKeys).set({ expiresAt: graceUntil }).where(eq(apiKeys.id, keyId)).returning()` — this is the grace-window expiry; `findByHash` already filters `expiresAt > now` so the old key stops authenticating exactly when the grace window closes (D-14).
    7. Return `{ oldKey, newKey }`.

    Wrap the whole thing in the same try/catch + `StorageError` re-throw pattern used by `create()` / `revoke()`.
